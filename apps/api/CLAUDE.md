# apps/api — Hono API on Cloudflare Workers

Thin, typed API server. Hono + Zod for routing and validation, Drizzle ORM over Cloudflare D1 for data, R2 for blob storage, Clerk for auth.

## Commands

```sh
bun run dev              # wrangler dev (local Workers runtime)
bun run build            # wrangler deploy --dry-run --outdir dist
bun run deploy           # wrangler deploy (production)
bun run test             # vitest run
bun run typecheck        # tsc --noEmit
bun run db:generate      # drizzle-kit generate (create migration files)
bun run db:migrate       # wrangler d1 migrations apply (run against local D1)
```

## Directory Structure

```
src/
  index.ts               Hono app entry — mounts all route groups, exports AppType
  types.ts               AppEnv (Bindings + Variables), AuthUser interface
  routes/
    canvas.ts            Canvas CRUD, save, thumbnail, favorite
    assistant.ts         AI assistant chat + run endpoints
    user.ts              User profile
    waitlist.ts          Waitlist signup
  middleware/
    auth.ts              Clerk token verification → requireAuth middleware
  lib/
    db/
      client.ts          createDb() — Drizzle client from D1 binding
      schema.ts          Drizzle table definitions + relations
    storage/
      canvas-storage.ts  R2 operations for canvas blobs and thumbnails
    assistant/
      service.ts         AI service layer
      parsing.ts         Prompt/response parsing helpers
    auth/
      build-auth-user.ts Convert Clerk user → AuthUser
      sync-user.ts       Upsert authenticated user into D1
```

## Route Authoring Pattern

Every route file follows this structure:

```ts
import { Hono } from 'hono';
import { zValidator } from '@hono/zod-validator';
import { requireAuth } from '../middleware/auth';
import type { AppEnv } from '../types';

export const fooRoutes = new Hono<AppEnv>()
  .use(requireAuth)              // auth middleware first
  .get('/list',
    zValidator('query', schema), // Zod validation on input
    async (c) => {
      const data = c.req.valid('query');
      const user = c.get('user');
      const db = createDb(c.env.DB);
      // ... query logic ...
      return c.json(result);     // typed response
    }
  );
```

Rules:
- Always validate inputs with `zValidator` using schemas from `@ai-canvas/shared/schemas`
- Always access the authenticated user via `c.get('user')` (set by `requireAuth`)
- Always create the DB client via `createDb(c.env.DB)` — do not cache across requests
- Return typed JSON responses. Use appropriate HTTP status codes (201 for create, 409 for conflict, 404 for not found).
- Mount route groups in `src/index.ts` via `.route('/api/prefix', fooRoutes)`

## Environment Bindings

Defined in `src/types.ts` as `AppEnv.Bindings`:

| Binding | Type | Purpose |
|---|---|---|
| `DB` | D1Database | Drizzle ORM target |
| `R2` | R2Bucket | Canvas blobs, thumbnails, assets |
| `CLERK_SECRET_KEY` | string | Auth token verification |
| `ANTHROPIC_API_KEY` | string? | Claude API access |
| `OPENROUTER_API_KEY` | string? | Multi-model routing |
| `VECTORIZE_ASSET_URL` | string? | Asset vectorization service |

## Database

- **ORM:** Drizzle with SQLite dialect (D1-native)
- **Schema:** `src/lib/db/schema.ts` — all tables, indexes, relations
- **Migrations:** `drizzle/` directory, generated by `drizzle-kit generate`
- **Column conventions:** `text` for strings, `integer` with `mode: 'timestamp'` for dates, `integer` with `mode: 'boolean'` for booleans
- **Naming:** snake_case for SQL columns, camelCase in Drizzle schema definitions
- **JSON columns:** Stored as `text` with `Json` suffix (e.g., `contextSnapshotJson`). Parse/serialize in application code, not the ORM.

Tables:
- `users` — synced from Clerk on auth
- `canvases` — canvas metadata (blob data in R2)
- `canvas_versions` — version snapshots
- `canvas_shares` — sharing permissions
- `waitlist_subscriptions` — waitlist signups
- `assistant_threads` — chat threads per canvas
- `assistant_runs` — individual AI runs with status tracking
- `assistant_run_events` — structured event log per run
- `assistant_tasks` — sub-tasks within a run
- `assistant_artifacts` — generated artifacts (mermaid, markdown, images, etc.)

## Storage

Canvas scene data and thumbnails are stored in R2, not D1. The database stores only metadata and an `r2Key` pointer.

- `canvas-storage.ts` provides `saveCanvasToR2`, `loadCanvasFromR2`, `saveThumbnailToR2`, `loadThumbnailFromR2`, `deleteCanvasFromR2`
- R2 keys are scoped by user ID and canvas ID

## Auth Flow

1. Frontend sends Clerk session token in `Authorization: Bearer <token>` header
2. `requireAuth` middleware verifies the token with Clerk's backend SDK
3. Clerk user is converted to `AuthUser` via `buildAuthUser`
4. User is upserted into the local `users` table via `syncAuthenticatedUser`
5. `AuthUser` is set on context via `c.set('user', user)` for downstream handlers

## Testing

Colocated with source (`.test.ts` next to the file). Pure helpers, parsing logic, and auth utilities are unit tested. Route handlers can be tested by constructing Hono request contexts.

## Type Safety Chain

The full type safety chain from API to frontend:

```
Zod schema (shared)  →  zValidator (api route)  →  AppType export (api index)  →  hc<AppType> (web client)
```

The frontend imports `AppType` from `@ai-canvas/api` as a dev dependency. No codegen, no manual type duplication.
